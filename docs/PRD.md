# Product Requirements Document — i-can-this-day

**URL family:** `https://hoy-puedo.dev.grupoalvs.com/<event-code>/`
**Status:** Draft · v0.1
**Date:** 2026-05-22

---

## 1. Product Summary

**i-can-this-day** is a lightweight, link-shareable event-coordination tool aimed at casual Spanish-speaking social groups. An organizer proposes a short list of candidate days for a shared event; invitees open the link and mark which day-parts (Mañana / Tarde / Noche) they can attend. The result is a visual calendar where the organizer — and every participant — can see consensus at a glance, without having to read through a thread of voice notes or emoji responses. No account setup is required beyond a Google login; no installation; one URL does everything.

The product is intentionally narrow: it does not manage RSVPs, seat limits, locations, or notifications. Its only job is to answer the question "¿qué día y a qué hora puede la mayoría?" as fast as possible from a phone screen.

---

## 2. Problem & Motivation

Coordinating small-group events over WhatsApp or Instagram DMs is chaotic: responses are buried in unrelated conversations, and tracking who said what for each day is manual work. The typical flow — "¿quién puede el sábado?", a flood of voice notes, then an informal tally — is slow, error-prone, and invisible to whoever joins the thread late. Existing scheduling tools such as Doodle or When2meet are either overly corporate, visually dense, or locale-mismatched for a casual Spanish-speaking audience. We want a single screen that any participant can open from a phone, fill in under sixty seconds, and share back in the same chat — with an instant visual signal of which day has the most traction. The app should feel like a natural extension of a WhatsApp conversation, not a productivity dashboard.

## 3. Core Concepts & Glossary

### 3.1 Entities

| Term | Definition |
|---|---|
| **Event** | A named scheduling proposal created by an organizer. It has a set of candidate days, a unique event code, and a collection of participations from users. An event has no defined end state in v1 — there is no "closed" or "finalized" status. |
| **Event code** | A short, opaque alphanumeric string embedded in the event URL (e.g., `HOI35L145KK`, `BAS529G45K3`). It is the sole access key to the event: any user who knows the URL can view the event. The code is generated by the app at creation time; organizers do not choose it. |
| **Candidate day** | A specific calendar date that the organizer has marked as eligible for the event. Only candidate days are interactive in the event calendar; all other days are greyed out. An organizer may mark any number of candidate days; zero is not a valid event (creation requires at least one). |
| **Day-part** | One of three named segments of a day: **Mañana**, **Tarde**, **Noche**. Day-parts divide a candidate day into sub-windows so participants can express partial availability (e.g., "I can do afternoon but not evening"). Exact local-time bands (e.g., 08:00–13:00) are cosmetic labels deferred to an ADR; the product model treats them as ordered, named slots. |
| **Participation** | A user's declaration that they can attend a specific (candidate day, day-part) cell within a specific event — colloquially "Yo puedo!". A single user may hold zero, one, two, or three participations per candidate day (one per day-part). Participations are mutable: a user can add, change, or implicitly remove them by saving a modal with fewer day-parts selected. |
| **Organizer** | The authenticated user who created the event. Distinguished from other participants by having additional edit permissions on the event's candidate days. Every event has exactly one organizer; there is no co-organizer or transfer-of-ownership concept in v1. |
| **Participant** | Any authenticated user who has declared at least one participation in the event. Participants are identified by their Google profile (name + avatar). Any user with the event link can become a participant with no explicit invitation required. |

### 3.2 Key Relationships

- One **Event** → zero or more **Candidate days**.
- One **Candidate day** → exactly three **Day-parts** (Mañana, Tarde, Noche).
- One **Participant** × one **Candidate day** → zero to three **Participations** (one per day-part).
- One **Organizer** → one or more **Events** (they may create multiple events over time).

### 3.3 Event Lifecycle

In v1, events do not have an explicit lifecycle state machine. The informal lifecycle is:

1. **Draft:** organizer selects candidate days on the creation form (not yet persisted).
2. **Active:** event is saved, code is generated, link is shared, participations are collected.
3. **Past (informal):** all candidate days are in the past. The event remains fully readable but the greyed-out styling signals it is historical. No write access is blocked — the data model does not enforce a hard close.

A formal "close voting" or "archive" action is out of scope for v1.

The absence of a defined lifecycle state means the app should never show a user a message like "this event is closed" or "voting has ended". The greyed-out visual treatment on the landing page and on individual days in the calendar is the only signal that a date is in the past.

## 4. Primary Surfaces / Screens

### 4.1 Landing Page

The landing page has a persistent header containing a single action button: **"Crear nuevo evento"**. The body lists every event the viewer has previously interacted with (created or participated in), sorted by date (most recent first). Future events render as clickable links leading to the event view. Past events render greyed-out and are not clickable (they remain visible for reference but cannot receive new participations). An anonymous viewer (not logged in) sees the same layout with an empty list and the create button; clicking **"Crear nuevo evento"** triggers a Google login prompt before proceeding to the creation form. A returning logged-in user sees their history immediately on load, with no loading gate or splash screen.

The header with the **"Crear nuevo evento"** button persists on all screens, not just the landing page, so participants can pivot to creating a new event from any context within the app (as illustrated in Historia: Claudia).

### 4.2 Event View (`/<event-code>/`)

The event view is a calendar grid for the relevant month(s). A month-navigation control at the top lets the user move between months. Each day occupies a soft, rounded square with the date number in the top-left corner. Days that are NOT candidate days render greyed-out and are non-interactive — no hover, no tap response. Candidate days show inside the square a summary of current participations broken down by day-part (e.g., "Tarde: 7 · Noche: 4"). This summary is visible to anonymous viewers without login. Tapping a candidate day while unauthenticated triggers a Google login prompt; once authenticated, the day modal opens automatically without requiring the user to tap again.

The grid layout should feel like a native calendar app, not a data table. Days from outside the current month (leading/trailing cells needed to fill a 7-column grid) are displayed dimmed and are non-interactive. There is no month title ambiguity: the navigation shows the current month name and year prominently.

### 4.3 Day Modal (inside Event View)

Opening a candidate day reveals a modal overlay. The modal shows:

- The date and event name as a modal header for orientation.
- Avatars of all participants who have declared "Yo puedo!" for that day, grouped under **Mañana / Tarde / Noche** headings. A day-part heading is visible even if zero participants have selected it, so the user understands all options.
- The current user's own participation state for the day (pre-filled with their last saved choices if they have interacted with this day before; all toggled off otherwise).
- Three toggles — one per day-part — that the user can flip independently of each other.
- A shortcut button **"¡Yo puedo todo el día!"** that activates all three day-parts simultaneously. If all three are already active, this button has no visible effect (idempotent).
- Two footer actions: **Cancelar** (discard any local changes and close the modal, leaving the persisted state unchanged) and **Guardar** (persist the current selection).

After a successful save, the modal closes and the calendar cell for that day updates immediately to reflect the new counts. If the save fails (network error or expired session), the modal remains open with an inline error message; the user is not silently dropped back to the calendar with stale data.

### 4.4 Create-Event Form

The creation form reuses the same calendar component as the event view, operating in **organizer mode**: the organizer taps days to toggle them as candidate days (tapping a day again deselects it). The interaction is identical to the event view's calendar but the semantic is different — days are being marked as "open for voting", not as personal availability. There is a minimal event metadata field (name/title — exact fields TBD in a later design pass). The form should default to the current or next calendar month.

When the organizer submits, the app mints a new event code and presents a confirmation screen that shows:

- The full shareable URL `https://<host>/<NEW-CODE>/`
- A **Copiar** button that copies the URL to the clipboard
- A direct link to open the new event view immediately

The app sends no invitations, notifications, or messages of any kind. The organizer is solely responsible for distributing the link through their chosen channels (WhatsApp, Instagram, etc.).

## 5. Authentication

Google login is the only supported authentication method. Email/password login is explicitly out of scope for v1. The following access rules apply:

- **Without login:** a user may open any event URL and see the full calendar with participation counts per candidate-day cell. No write interaction is permitted.
- **With login required:** declaring or editing participations, and creating new events, both require an active Google session.
- **Login trigger:** the first interactive action that requires authentication (tapping a candidate day, or clicking "Crear nuevo evento") presents a Google login prompt inline — either as an overlay or a redirect-and-return flow. The user must return to the same event/page context after authenticating, not to a generic home screen.
- **Session lifetime:** sessions are long-lived and persist across browser restarts (cookie/storage backed). Sessions silently expire after a configurable period of inactivity. The exact TTL is deferred to an ADR.
- **Identity:** the user's Google profile name and avatar are used for display in the day modal participant list. No additional profile fields are collected in v1.
- **Logout:** a logout action must exist and must be accessible from the app (exact placement = design decision), but it need not be prominently surfaced — casual users typically do not log out explicitly, relying on session expiry instead.

## 6. Authorization Model

Three roles emerge implicitly from product behavior:

| Role | How acquired | Permissions |
|---|---|---|
| **Anonymous viewer** | Default; no login needed | View calendar, view participation counts. No writes. |
| **Participant** | Any authenticated user who opens a valid event link | Declare and edit their OWN participations on any candidate day of that event. Cannot modify other users' participations or the event's candidate days. |
| **Organizer** | The authenticated user who created the event | All participant permissions, plus ability to add or remove candidate days after event creation (exact edit semantics = ADR). |

**Possession-of-link is the access token.** There is no invitation list, no email whitelist, no moderator approval flow in v1. This is a deliberate design decision for casual social use; it is not a deficiency. If the organizer wants to restrict access, they simply do not share the link — the app does not provide enforcement tooling for this.

**Cross-event isolation:** a participant's role is scoped to a specific event. Being a participant in event A grants no permissions in event B, even if the same user created both events.

## 8. Non-Functional Requirements

### NFR-1 — Mobile-First Responsive Design

The primary user interaction model is opening a WhatsApp link on a smartphone. All primary actions must be usable on a screen as narrow as 375 px without horizontal scrolling or pinching. The calendar grid must fit entirely within the viewport width on a standard phone; day squares may be small but must have touch targets of at least 44 × 44 px per platform guidelines. Desktop and tablet layouts are secondary concerns but must not be broken; the design scales gracefully upward.

### NFR-2 — Accessibility (WCAG 2.2 AA Target)

- Color must not be the sole indicator of state. Greyed-out days need a visual treatment (reduced contrast AND a structural difference, not just color).
- All interactive elements must meet a minimum contrast ratio of 4.5:1 against their background.
- Modals must trap focus correctly: Tab cycles within the modal; Escape closes the modal; focus returns to the triggering element on close.
- The calendar grid must be keyboard-navigable (arrow keys move between days; Enter/Space opens the modal for candidate days).
- Avatar images in the modal must have descriptive `alt` text (participant name).

### NFR-3 — Spanish UI as Default

All user-facing copy ships in Spanish. Dates, months, and weekday labels follow the `es-AR` locale convention (Monday as first day of week; month names in lowercase). Internationalization hooks in the codebase are acceptable but are not a v1 deliverable; do not block development on i18n infrastructure.

### NFR-4 — Perceived Latency

Calendar cell interactions (opening the modal, toggling a day-part, pressing Guardar) must feel instantaneous. The target for optimistic UI feedback is under 200 ms from user action to visual state change. Network latency for the actual persistence call is a separate concern — the UI must update optimistically and only reconcile on server response.

### NFR-5 — URL Stability and Deep-Linking

The app must function correctly under the URL family `https://<host>/<event-code>/`. Navigating directly to a specific event URL must render the event view without a server-side redirect to a root path or a 404. URLs must be stable over the lifetime of the event (no code rotation, no expiry of the URL).

### NFR-6 — Data Persistence

Events and participations must survive indefinitely in v1. There is no scheduled cleanup, expiry timer, or garbage-collection job. A participant who declared "Yo puedo!" three months ago will still see their participation if they revisit the event link.

### NFR-7 — No Installation Required

The product is a web application accessible from any modern mobile browser with no download, no app store, and no native runtime. The target browser baseline is Safari on iOS 16+ and Chrome on Android 12+, plus current desktop versions of Chrome, Firefox, and Safari.

### NFR-8 — Session Resilience and Graceful Degradation

A user who leaves the app open for hours (e.g., a WhatsApp in-app browser tab left idle) must not silently lose a participation they already saved. If a session has expired by the time the user presses Guardar, the app must surface a clear re-authentication prompt rather than silently discarding the write. Stale-session handling must be explicit and recoverable, not a silent data loss.

## 9. User Stories

### Historia: Marcos

Marcos recibe un link "https://hoy-puedo.dev.grupoalvs.com/HOI35L145KK/", al abrirlo puede ver un calendario de este mes, con todos los días grises excepto el 29, 30 y 31 de mayo. Alcanza a ver que el 30 de mayo a la tarde tiene 7 "yo puedo!", a la noche tiene 4 "yo puedo!", y los otros días no le interesan: así es que hace click en el día 30. Entonces la app le solicita ingresar su login de google, marcos usa msunombre@gmail.com y continúa con la app. Ahora, se abre un pop up modal donde Marcos ve quiénes son los que han dicho "yo puedo", y ve los avatares de todos. La mayoría acumulados a la tarde, y algunos repeitdos a la noche y la mañana. Marcos puede a la Mañana y a la Noche, así es que llena esos valores. Ahora puede cancelar (limpiar) o guardar, y prefiere guardar. Marcos ve ahora que el día 30 ha cambiado, ya están sus dos entradas agregadas.

**Criterios de aceptación:**

- Al abrir el link sin sesión activa, los días 29, 30 y 31 de mayo se muestran como candidatos interactivos; el resto del mes aparece gris y no responde a clics.
- Las celdas del 30 de mayo muestran "Tarde: 7" y "Noche: 4" (conteos visibles sin login).
- Al hacer clic en el día 30, la app presenta el flujo de Google login antes de abrir el modal.
- Tras autenticarse, el modal muestra avatares agrupados por Mañana / Tarde / Noche con los participantes previos.
- El usuario puede seleccionar Mañana y Noche de forma independiente y presionar Guardar sin error.
- Después de guardar, la celda del día 30 en el calendario refleja inmediatamente los conteos actualizados que incluyen las dos participaciones de Marcos.

- Las tres participaciones de Patricia (una por día, tres day-parts cada una) quedan persistidas y son visibles en los modales respectivos al reabrirlos.
- Los conteos de cada celda reflejan correctamente las tres participaciones de Patricia además de las preexistentes de otros usuarios.

## 10. Success Metrics (v1 Baseline)

These metrics are not hard SLAs but define what "working" looks like at launch. They should be measurable from server-side logs or minimal analytics without requiring a full analytics platform.

| Metric | Target | Notes |
|---|---|---|
| **Event creation completion rate** | > 70% of users who open the creation form complete it and generate a link | Measures whether the form is simple enough. |
| **Participation declaration rate** | > 60% of link recipients who open the event tap at least one candidate day | Measures whether the UI drives action. |
| **Mobile share** | > 80% of sessions originate from mobile browsers | Validates the mobile-first design priority. |
| **Same-day engagement** | > 50% of participations happen within 24 h of the link being shared | Validates the "quick response" use case. |
| **Zero participation events** | < 20% of events reach their last candidate day with zero participations | A proxy for link distribution success (outside our control) and UI friction (within our control). |

## 11. Out of Scope (v1)

The following capabilities are explicitly deferred and must not be implemented in v1, even if technically simple. Each item includes a brief rationale so that future scope discussions start from context rather than from scratch.

1. **Notifications from the app** — No email, SMS, or push notifications triggered by the app when someone adds a participation or when a new event is created. Rationale: the organizer distributes the link through their existing channels (WhatsApp, Instagram), so participants are already in a communication context. In-app notifications add infrastructure complexity without proportional value for v1.

2. **WhatsApp / social-media share integration** — The app provides a URL and a **Copiar** button; it does not call social-media share APIs, pre-populate WhatsApp messages, or open OS-level share sheets. Rationale: the organizer always knows better than the app which group or thread to share into. A generic share-to-WhatsApp button is a distraction.

3. **Invitee whitelists or seat limits** — No maximum participant count, no approved-user list, no waiting-room flow. Rationale: the target use case is a small friend group; the social context of the shared link provides sufficient implicit access control. Adding whitelists introduces friction and admin work that is out of proportion for the use case.

4. **Recurring events** — Each event is a standalone, one-time scheduling proposal. A "repeat this event weekly" or "copy event" feature is deferred. Rationale: the data model and UX of recurring events (exception handling, individual-occurrence overrides) is significantly more complex than the v1 scope.

5. **Time-zone management beyond a single local zone** — The app operates in a single server-configured local time zone. Per-user time-zone display, UTC offset detection, or "convert to my local time" tooltips are out of scope. Rationale: all candidate users in v1 are in the same city/region. Cross-timezone scheduling is a different product category.

6. **Calendar export (.ics / iCal)** — No export to Google Calendar, Apple Calendar, Outlook, or any other calendar application. Rationale: an event-coordination poll is not a confirmed appointment; exporting to a calendar implies a finalized date, which is a later step the organizer handles outside the app.

7. **Comment threads or freeform discussion** — No text comments, free-form notes, or discussion threads attached to days or events. Rationale: the app's value proposition is the visual grid. Adding a chat layer recreates the very problem (buried text in a thread) that the app solves.

8. **Anonymous participation (no Google login)** — "Yo puedo!" declarations require a Google login. No guest/anonymous voting where users self-identify by typing their name. Rationale: avatar-based identification of participants is a core UX element of the day modal; anonymous participation would degrade that experience and create spoofing opportunities.

9. **Mobile native apps (iOS / Android)** — The product ships as a web application only. Native apps are not in scope for v1. Rationale: the primary sharing vector (WhatsApp link) opens in a mobile browser or in-app browser, where a web application works natively. A native app would require app store distribution, which adds friction to the very first interaction.

10. **Event deletion or archiving by the organizer** — Past events persist indefinitely in v1. There is no "delete event", "archive", or "close voting" action. Rationale: simplifies the data model and avoids edge cases around deletion of events with active participants. Re-evaluate in v2 once the event lifecycle is better understood.

## 11. Open Questions (for ADR Phase)

- **Day-part time bands:** What are the exact local-time ranges for Mañana, Tarde, and Noche (e.g., 08:00–12:00 / 12:00–19:00 / 19:00–00:00)? Are these ranges shown to users or are they purely cosmetic labels that help coordinate socially?
- **Session inactivity TTL:** How long before a Google session silently expires? Typical range is 7–30 days. What is the re-authentication UX when expiry happens mid-modal-interaction?
- **Organizer edit semantics — removal:** If the organizer removes a candidate day that already has participations from other users, what happens to those participation records? Options: hard-delete, soft-delete (hidden but recoverable), or block the removal if participations exist.
- **Organizer edit semantics — addition:** When a new candidate day is added to a published event, do existing participants see a visible change indicator (e.g., a "new!" badge)? Or does the calendar simply show the new day as an interactive cell with zero counts?
- **Event-code generation:** What is the alphabet (alphanumeric? uppercase-only?), minimum length, and collision detection strategy? The examples (`HOI35L145KK`, `BAS529G45K3`) suggest 11-character uppercase alphanumeric — should this be codified?
- **Past-event visibility on landing page:** Are past events listed for all logged-in users who participated, or only for the organizer? Are they visible to anonymous viewers at all?
- **Multi-month events — default view:** If an organizer marks candidate days across two calendar months, which month does the event view open to by default? Options: earliest candidate month, the current calendar month, the organizer's specified "main" month.
- **Concurrent save collisions:** If two participants save participations for the same (event, day, day-part) cell within milliseconds of each other, is optimistic last-write-wins acceptable, or do we need server-side atomic upsert semantics?
- **Account identity stability:** The app associates participations with a Google identity. If a user signs in with a different Google account, their previous participations are invisible. Is this acceptable, or should we provide a way to merge/transfer participations?
- **Past-event cutoff definition:** What makes a day "past"? The calendar day itself (midnight local time), or the last day-part end time (e.g., midnight after the "Noche" window ends)? This affects the greyed-out styling on the landing page and whether interactions on the current day are allowed.
- **Event name field:** The creation form includes a name/title field (mentioned in Section 4.4). Is this field required or optional? Is it displayed in the event view header, the modal header, and/or the landing page event list?

## 12. Design Principles

These principles are not requirements but they guide prioritization when tradeoffs arise. Engineering and design should use them to resolve ambiguity without escalating every decision.

### 12.1 One action, one screen

The app should never require the user to navigate across multiple screens to complete a single logical action. Declaring availability for a day is one action; it happens entirely within the day modal. Creating an event is one form; it opens and submits on the same screen.

### 12.2 Visible state at all times

The calendar is the source of truth. Any change to a participation must be immediately reflected in the calendar cell. A user should never need to reload the page to see whether their save was successful.

### 12.3 No data loss by surprise

If the app cannot save (network error, expired session), it must tell the user clearly and preserve their unsaved input. Silent failures that appear successful but do not persist are unacceptable.

### 12.4 WhatsApp-first entry point

The most common first interaction is: receive link in WhatsApp → open in in-app browser → see the calendar → tap a day → log in → fill in availability → close. The design must be optimized for this flow. Long onboarding, pop-ups asking for notifications, or any step between "receive link" and "see the calendar" must be eliminated.

### 12.5 The calendar IS the product

The visual calendar is not a navigation element — it IS the product. It should be the largest, most prominent element on any event-related screen. Participation counts should be readable at a glance from a phone held at arm's length.
